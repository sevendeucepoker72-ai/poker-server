# Canonical Features Manifest — poker-server (Railway socket server)
#
# PURPOSE: prevent feature regressions on deploy. The deploy script
# (when wired) greps the BUILT bundle (dist/) and/or src/ for every
# non-comment/non-blank line in this file and ABORTS the deploy if any
# are missing. This is the same pattern as the mono-repo apps:
# - apps/web/canonical-features.txt        (player web)
# - apps/web/poker-admin/canonical-features.txt (admin)
# - apps/marketing/canonical-features.txt (marketing)
#
# NOTE: 2026-05-12 — this manifest is currently READ-ONLY INVENTORY.
# poker-server's Railway deploy script does NOT yet grep against it. The
# file is committed now to establish the lock surface; the deploy-side
# enforcement will be wired in a follow-up. Per CLAUDE.md's "three layers
# of locks" rule, having the manifest in place is layer #1 (the contract);
# the grep step is layer #2 (enforcement).
#
# RULES:
# - One token per line
# - Tokens matched as plain substrings (literal grep -F, not regex)
# - Lines starting with `#` are comments. Blanks ignored.
# - Use string literals (CSS class names, log tags, event-name strings,
#   route paths) that survive Terser minification. Function names get
#   mangled.
# - When you ship a NEW feature you don't want to regress: append the
#   identifying tokens in the SAME commit as the feature.
# - When you legitimately retire a feature: delete the matching tokens
#   in the same commit. The diff is the audit trail.
